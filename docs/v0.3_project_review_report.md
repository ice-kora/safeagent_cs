## 3. Overall Conclusion

SafeAgent-CS v0.3 已经不是简单 Demo，而是一个具备较强工程化雏形的受控客服 Agent 执行平台原型。

它的核心价值在于：

- 没有把 LLM、Planner 或 RAG 放在最终执行裁决位置。
- 明确保留 Validator、PolicyService、ToolGateway、PendingActionService、FailureHandler 等确定性安全边界。
- 同时支持 manual、workflow-style、real LangGraph chat workflow 三条编排路径。
- 用 Safety Regression 和大量 pytest 回归测试压住安全语义。
- 对 checkpoint / resume 没有贸然开启真实执行，而是先补 snapshot、协议、dry-run、矩阵和幂等边界准备。

客观判断：

- 作为作品集或技术展示项目：质量较高，亮点明确。
- 作为企业落地原型：方向正确，安全边界意识强。
- 作为生产系统：仍有明显距离，主要缺 AuthN/AuthZ、真实业务系统适配、多租户隔离、持久化幂等事实源、真实 LLM 风险治理和运维控制台。

总体评分：8.1 / 10。

## 4. Architecture Review

当前架构分层清晰：

```text
API Layer
  -> Planning Layer
  -> Guard / Validation Layer
  -> Policy Kernel
  -> Execution Gateway
  -> Workflow / Trace / Evaluation
```

架构优点：

- API 入口清楚：`/api/chat` 处理首次请求，`/api/confirm` 处理二次确认恢复。
- Planner 与执行解耦：Planner 只生成候选 ActionPlan。
- 权限裁决集中：PolicyService 是权限与风险判断核心。
- 工具调用集中：ToolGateway 是唯一主链路工具入口。
- 编排层可替换：manual 与 workflow 路径共存，LangGraph 只做编排。
- 可观测性基础完整：run、trace、tool logs、failure logs 等链路齐备。

架构不足：

- PolicyService 仍是集中式规则入口，未来规则增多后需要 registry / rule class 化。
- RepositoryService 当前仍是 mock 数据源，距离真实数据权限模型较远。
- workflow 与 manual 双轨虽然测试覆盖较好，但长期维护仍有重复语义漂移风险。
- 真实生产配置、环境隔离、密钥管理尚未形成体系。

## 5. Safety Boundary Review

安全边界总体较完整。

已确认的安全边界：

- ActionPlanValidator 负责结构合法性，不判断真实业务权限。
- PolicyService 负责真实权限和风险裁决，不调用工具。
- ToolGateway 负责工具白名单、路由、日志、脱敏，不做权限裁决。
- FailureHandler 重试仍通过 ToolGateway，不直接调用 mock tool。
- CONFIRM_REQUIRED 只创建 pending action，不直接执行工具。
- DENY / HUMAN_REQUIRED 分支不进入 ToolGateway。
- `/api/confirm` 执行前重新复核 PolicyService。
- RAG 只服务 `knowledge_tool.query_policy`，不参与订单权限裁决。
- LLM 相关代码目前停留在契约和 Guard 层，没有真实 LLM 接入。

静态检查备注：

- `app/tools/registry.py` 内部直接调用具体 mock tool，这是 ToolGateway 路由所需的代码级 registry，不属于主链路绕过。
- `app/evaluation/rag_runner.py` 直接调用 `knowledge_tool.query_policy`，这是离线评测路径，不是 `/api/chat` 执行链路。
- 测试文件中存在 spy/fake gateway 和直接工具测试，属于验证需要。

主要风险：

- 目前尚未接入真实身份认证和授权，`user_id` 仍来自请求体或测试上下文。
- 多租户隔离停留在策略规则与 mock context 层，未达到生产级。
- 工具幂等仍处于 readiness，真实重复执行防护还不完整。

## 6. Workflow / LangGraph Review

当前支持：

- manual 主链路。
- LangGraph-style 轻量 workflow。
- real LangGraph chat workflow engine。
- workflow mode / workflow engine 配置开关。

检查结论：

- real LangGraph 只用于 chat workflow 编排。
- LangGraph 节点复用现有 service，不重新实现 PolicyService、ToolGateway、FailureHandler。
- 默认配置仍是 manual / style，不默认启用真实 LangGraph。
- workflow mode 和 engine 灰度逻辑清楚。
- manual 主链路仍可用，并有测试保护。

优点：

- 编排层职责控制得较好。
- 与手写流程保持兼容响应结构。
- 通过 safety regression 和 LangGraph 专项测试验证分支语义。

不足：

- LangGraph 目前主要是编排迁移，尚未充分利用 interrupt / checkpoint 能力。
- confirm workflow 已灰度接入，但 real LangGraph 重点仍在 chat workflow。
- 后续如果同时维护 manual/style/langgraph，必须防止三套路径语义漂移。

## 7. RAG Review

当前 RAG 设计定位正确：只服务公开政策查询。

已具备：

- 本地轻量 RAG 模块。
- policy corpus。
- chunk / retriever / reranker / safety。
- sources / citation。
- RAG evaluation。
- manual 与 workflow 路径的 RAG 回归测试。

优点：

- 没有把 RAG 设计成权限系统。
- RAG 输出走 ToolGateway 主链路。
- 有 no-answer / citation quality 相关测试。

不足：

- 仍是本地轻量实现，不是向量库。
- 检索算法偏规则和关键词，复杂语义召回能力有限。
- 语料规模小，无法代表真实企业知识库治理难度。
- 还缺文档版本、知识源可信等级、召回监控和在线评估。

## 8. Checkpoint / Resume Readiness Review

当前 checkpoint/resume 的处理是谨慎的，符合高风险系统应有节奏。

已完成：

- `LangGraphCheckpointSnapshot`。
- `checkpoint.snapshot.v1` schema version。
- JSON-safe snapshot。
- checkpoint / resume protocol。
- `InMemoryCheckpointStore`。
- resume dry-run。
- tool resume readiness。
- pending action resume readiness。
- resume matrix。

重要边界：

- 未启用真实 checkpointer。
- 未使用 MemorySaver / SQLiteSaver。
- 未新增 `/api/resume`。
- dry-run 不调用 ToolGateway。
- dry-run 不创建 pending action。
- side-effect node 默认禁止真实 resume。
- tool resume 要求 `tool_call_id` / `idempotency_key`。
- pending resume 要求 `pending_action_id`。

不足：

- InMemoryCheckpointStore 不能跨进程、跨重启。
- tool_call_logs 尚未形成完整 resume 所需的 tool_call_id 协议。
- pending action 缺 event sourcing，状态历史不足。
- 真实 resume 所需的 run 生命周期恢复策略尚未实现。

## 14. Interview Risk Points

### 1. 为什么没有真实 LLM？

回答建议：当前阶段优先建立可验证安全边界。LLM 接入已通过 LLM schema、LLMOutputGuard、LLMResponseGuard 做好契约准备，但不会让 LLM 绕过 Validator、PolicyService 和 ToolGateway。

### 2. 为什么 checkpoint 只做 readiness？

回答建议：工具调用和 pending action 都是副作用边界。没有幂等事实源就打开 resume，可能造成重复执行，所以先做 snapshot、matrix、dry-run 和 resume policy。

### 3. 为什么 RAG 不是向量库？

回答建议：当前重点是证明 RAG 在受控 Agent 中的边界：只服务公开政策查询，不参与权限裁决。真实向量库属于后续替换层，不影响安全内核设计。

### 4. PolicyService 和 Validator 的边界是什么？

回答建议：Validator 判断计划结构是否合法；PolicyService 判断当前用户是否能做、风险多高、是否需要确认或人工。

### 5. ToolGateway 如何防止重复执行？

回答建议：当前已集中工具入口、tool_call_logs 和 resume readiness。下一步需要补 `tool_call_id`、`idempotency_key` 持久化和查询协议。

### 6. confirm 为什么要重新复核？

回答建议：confirm 是新的用户输入和新的 run。订单状态、权限、风险都可能变化，因此必须复核 PolicyService，而不是直接执行旧计划。

### 7. 如何上生产？

回答建议：需要补真实 AuthN/AuthZ、多租户隔离、真实业务系统 adapter、工具幂等持久化、pending_action_events、观测控制台、告警和安全审计。

## 16. Next-Step Priority

建议优先级：

1. Tool idempotency persistence：补齐 `tool_call_id`、`idempotency_key`、attempt 和查询协议。
2. PendingAction event sourcing：新增 pending_action_events，记录状态流转历史。
3. AuthN/AuthZ hardening：明确 actor、role、customer、tenant 边界。
4. Multi-tenant isolation：把 tenant 约束贯穿 API、Repository、Policy、Logs。
5. Real LLM Provider Adapter：在 Guard 与 fallback 完整前提下接入。
6. Real vector retrieval：替换本地轻量 RAG，但仍只用于知识查询。
7. Real checkpoint：在工具幂等和 pending event 完成后接 checkpointer。
8. Frontend console：展示 run、trace、policy logs、tool logs、failure logs。

不建议下一步：

- 直接实现 `/api/resume`。
- 直接让 LLM 调工具。
- 直接接 MCP 绕过 ToolGateway。
- 在无幂等事实源前开启工具节点恢复。
