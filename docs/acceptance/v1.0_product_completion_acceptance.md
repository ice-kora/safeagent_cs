## Completion Scope

SafeAgent-CS v1.0 is frozen as a local demo-ready customer service safety agent. The accepted scope includes Customer Chat, rule/workflow/LangGraph orchestration, ToolGateway safety boundaries, PostgreSQL demo data, checkpoint/resume, observability, mock MCP and Feishu skeleton integration, and real RAG backed by Milvus and BGE-M3.

No git commit, push, or tag was performed during this acceptance pass.

## Customer Chat

Customer Chat is accepted for the v1.0 demo flow.

- Primary endpoint: `/api/chat`
- Console path: `/console/`
- Supports order query, address-change confirmation, policy/RAG queries, complaint routing, no-answer handoff, and prompt-injection denial paths.
- RAG responses expose concise evidence in the chat response and detailed debug fields under `rag`.
- Debug fields include `query`, `retrieval_mode`, `embedding_model`, `vector_store`, evidence chunks, and per-evidence `score_detail`.

Verified Customer Chat RAG cases:

- Refund rule query: returned `policy_return_7d`, `policy_refund_sla`, `policy_after_sales_rules`.
- Address-change rule query: returned `policy_address_change`, `policy_shipped_address_change`, `policy_order_query`.
- Order-query rule query: returned `policy_order_query`, `policy_shipped_address_change`, `policy_logistics_query`.
- Complaint ticket rule query: returned `policy_complaint_ticket`, `policy_after_sales_rules`, `policy_after_sales_human`.
- No-answer query: returned `POLICY_NOT_FOUND`, no evidence, and human-handoff guidance.

## Real RAG

Real RAG is accepted for v1.0 demo.

- Vector store: Milvus
- Embedding provider: BGE-M3 via `FlagEmbedding`
- Default embedding model: `BAAI/bge-m3`
- Retrieval mode: hybrid
- Dense score source: Milvus vector search
- Keyword score source: local lexical scoring
- Hybrid formula: `dense_weight * dense_score + keyword_weight * keyword_score`
- Default weights: dense `0.7`, keyword `0.3`
- RAG remains evidence-only and does not participate in `PolicyDecision`.
- Tool execution remains exclusively through `ToolGateway`.

## Prompt Injection

Prompt Injection behavior is accepted for v1.0.

Validated behavior:

- Prompt-injection intent is classified before normal business intents.
- Dangerous instructions such as ignoring safety rules, reading system prompt, exporting users, or bypassing policy are denied.
- LLM output, when enabled, remains subject to output guard, ActionPlan validation, PolicyService, and ToolGateway boundaries.
- Console and observability payloads sanitize system prompt, API keys, tokens, full phone numbers, full addresses, and payment information.

Coverage is included in the full pytest suite, especially workflow and LLM safety regression tests.

## Checkpoint And Resume

Checkpoint/Resume is accepted for v1.0.

Validated behavior:

- Medium-risk address changes create `pending_action` and checkpoint records.
- Resume does not execute tools directly.
- Resume returns the flow to confirmation and policy validation.
- Confirmed actions execute through `ToolGateway`.
- Cancelled and expired states are represented in PostgreSQL demo seed.

Coverage is included in checkpoint, pending-action, confirm, and workflow tests.

## Known Limitations

- This is a local demo acceptance, not a production deployment.
- BGE-M3 runs on CPU and may be slow on first load.
- Hugging Face unauthenticated requests may be rate-limited; local cache is recommended.
- Real RAG uses Milvus dense search plus local lexical reranking, not Milvus native sparse hybrid search.
- Feishu integration remains a skeleton.
- MCP mock remains optional and explicitly gated.
- Real external business adapters remain fail-closed.
- RAG is intentionally evidence-only and cannot make policy decisions or execute tools.
