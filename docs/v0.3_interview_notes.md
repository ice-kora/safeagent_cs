# SafeAgent-CS v0.3 Interview Notes

## 1. 30 秒介绍

SafeAgent-CS 是一个企业级受控客服 Agent 执行平台。它把 LLM、Planner、RAG 和工具调用放进可审计、可测试、可二次确认的安全主链路中。系统核心不是让模型直接操作业务，而是让模型只提出候选计划，最终是否执行由 Validator、PolicyService、ToolGateway、PendingActionService 和 FailureHandler 等确定性组件决定。

## 2. 2 分钟项目讲解

项目从客服场景出发，实现了 `/api/chat` 首次请求入口和 `/api/confirm` 二次确认入口。用户请求进入后，系统先做意图识别和 ActionPlan 生成，然后经过 ActionPlanValidator 做结构校验，再由 PolicyService 做权限和风险裁决。

如果裁决为 ALLOW，工具调用必须经过 ToolGateway；如果是 CONFIRM_REQUIRED，只创建 pending action，不直接执行；如果是 HUMAN_REQUIRED，返回人工处理；如果是 DENY，拒绝且不进入工具。

v0.3 进一步补齐了 workflow mode、真实 LangGraph chat workflow engine、RAG KnowledgeTool、Safety Regression、checkpoint snapshot 和 resume readiness。当前没有开启真实 resume，但已经具备 schema、矩阵、dry-run 和幂等边界准备。

## 3. 架构亮点

- 安全内核与流程编排分离。
- ToolGateway 是唯一工具入口。
- PolicyService 是最终权限与风险裁决点。
- Workflow 和 LangGraph 只负责编排，不复制业务规则。
- RAG 只做公开政策检索，不参与权限。
- Checkpoint / Resume 先做 readiness，不急于打开副作用恢复。

## 4. 安全亮点

- Validator 失败不进 Policy。
- Policy 不放行不进 ToolGateway。
- CONFIRM_REQUIRED 不直接执行。
- `/api/confirm` 必须重新复核 Policy。
- FailureHandler 重试仍走 ToolGateway。
- Trace 和日志脱敏。
- LLMResponseGuard 防止模型把 DENY 改写成成功。

## 5. LangGraph 亮点

LangGraph 在项目里是编排层：

- 把节点顺序和条件路由显式化。
- 保留现有 service 边界。
- 为 checkpoint / resume 做准备。
- 通过配置开关灰度启用。

它不负责权限、工具执行、pending action 状态管理和失败重试策略。

## 6. RAG 亮点

RAG KnowledgeTool 只服务公开知识查询：

- 本地轻量语料。
- citation sources。
- evaluation runner。
- 不触碰订单权限。

这避免把“检索到了什么”误当成“允许做什么”。

## 7. Checkpoint / Resume 亮点

当前 checkpoint/resume 只做 readiness：

- JSON-safe snapshot。
- schema version。
- node risk matrix。
- dry-run。
- tool resume idempotency readiness。
- pending action resume readiness。

真实 resume 暂不开启，因为工具调用和 pending action 都是副作用边界，必须先有幂等事实源。

## 8. 面试官可能追问与回答口径

### 1. 为什么不用 LLM 直接调用工具？

因为客服系统会触及订单、退款、地址等高风险动作。LLM 输出只能作为候选计划，必须经过 Validator、PolicyService 和 ToolGateway，避免越权、误执行和不可审计。

### 2. LangGraph 在项目里负责什么？

LangGraph 只负责流程编排和条件路由，不负责业务安全裁决。节点内部仍复用现有 service。

### 3. PolicyService 和 Validator 有什么区别？

Validator 管结构合法性，例如 action、tool_name、必要参数。PolicyService 管真实权限和风险，例如订单是否属于当前客户、租户是否一致、是否需要确认或人工。

### 4. ToolGateway 为什么必须是唯一入口？

因为工具调用需要统一白名单、参数脱敏、结果摘要、tool_call_logs 和重试边界。如果绕过 ToolGateway，就无法审计和控制副作用。

### 5. RAG 在项目里是不是权限系统？

不是。RAG 只检索公开政策知识，不决定用户是否能查订单、改地址或退款。权限仍由 PolicyService 裁决。

### 6. 为什么 confirm 要重新检查 Policy？

因为确认是新的用户输入和新的 run。原始 pending action 可能已经过期，订单状态也可能变化，所以必须复核。

### 7. 为什么 checkpoint/resume 不能直接打开？

因为恢复到工具或 pending action 附近可能造成重复工具调用或重复创建确认动作。必须先有 idempotency metadata、事件日志和恢复矩阵。

### 8. 如何避免工具重复执行？

通过 ToolGateway 统一入口、tool_call_logs、idempotency_key、tool_call_id 和 resume policy。当前只做到 readiness，真实执行恢复仍未开启。

### 9. 如果要接真实 LLM，怎么接？

通过 LLMIntentClassifier、LLMActionPlanner、LLMResponseGenerator 接入，但输出必须先经过 LLMOutputGuard / ActionPlanValidator / PolicyService / ToolGateway。最终回复还要过 LLMResponseGuard。

### 10. 如果要上生产，还缺什么？

缺真实 AuthN/AuthZ、多租户隔离、真实订单系统适配、持久化 checkpoint、tool idempotency persistence、pending_action_events、观测控制台、告警、限流和安全评审。
